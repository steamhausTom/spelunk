# TASK-023: Normalise error paths to relative and add dependency lock file

**Status**: Pending
**Wave**: 1 — Remediation
**Effort**: S — two discrete sub-items: (1) relativise error path strings in walk_repo; (2) generate and commit a pip-tools lock file
**Base Branch**: feature/wave-1-foundation
**Dependencies**: TASK-005

## Description

Two hygiene items surfaced by the security review:

**1. Relative error paths (SEC-F4.2)**

`walk_repo` currently appends absolute paths to `errors[]` (e.g. `"Permission denied: /home/user/secret-project/src/file.py"`). These strings flow into `meta.errors[]` in the final JSON output, which may be consumed by downstream pipelines or logged. Absolute paths leak directory structure to anything that receives the scan output.

Fix: before appending to `errors`, compute the path relative to the scan root using `path.relative_to(resolved_root)`. If `relative_to` raises (path is outside root — shouldn't happen in normal operation but must be handled), fall back to `path.name` only.

**2. Dependency lock file (SEC-F4.3)**

`pyproject.toml` declares open-ended version constraints. Without a committed lock file, `pip install` resolves to the latest satisfying version at install time — meaning the CVE status of transitive dependencies is non-deterministic across environments and CI runs.

Add a `requirements/` directory containing:
- `requirements.txt` — locked runtime dependencies generated by `pip-compile pyproject.toml`
- `requirements-dev.txt` — locked dev dependencies generated by `pip-compile pyproject.toml --extra dev`

Use `pip-tools` (`pip-compile`) to generate. Add `pip-tools` to the `[dev]` optional dependencies. Document the regeneration command in `CLAUDE.md` under Key Commands.

## Acceptance Criteria

- [ ] [eng] Given a `PermissionError` on a file inside the scan root · When `walk_repo` runs · Then the error string in the returned `errors` list contains a path relative to the scan root, not an absolute path
- [ ] [eng] Given a non-permission `OSError` on a file inside the scan root · When `walk_repo` runs · Then the error string contains a relative path, not an absolute path
- [ ] [eng] Given `pyproject.toml` · When `pip-compile pyproject.toml --output-file requirements/requirements.txt` is run · Then the command exits 0 and `requirements/requirements.txt` already matches (no drift)
- [ ] [qa] Given the repo root · When `ls requirements/` is run · Then both `requirements.txt` and `requirements-dev.txt` exist and are committed to version control

## Notes / Risks

- Changing from absolute to relative paths is a minor breaking change to the error string format. It does not affect the JSON schema shape (both are still strings) but it does change the string content. Update any existing test assertions that assert on the exact error string.
- `pip-tools` generates lock files from `pyproject.toml` directly. The generated files should include a header comment that specifies the command used to regenerate them, so future maintainers can update them.
- Add `requirements/*.txt` to `.gitignore` exclusions with a `!requirements/*.txt` override, or simply ensure `.gitignore` does not currently exclude `requirements/` (check before implementing).
- This is a Low/Informational severity finding. It does not block the Wave 1 PR but is included in the remediation wave to close all open security review items.
